**3 ADR 일관 패턴 발견**:

1. **"도메인 적정 트레이드오프"** 라는 표현이 3 ADR 모두에 등장 — 청첩장 도메인 (위협 모델 약함, 1회 사용, ~100명 풀) 의 특수성 명시 + **다른 도메인 fork 시 부적합** 경고
2. **거부 대안 명시 5~6 종** — 결정의 견고성 확보. 12주차 ADR 005 (다중 테마) 의 5 거부 대안 패턴 일관 유지
3. **Console 토글로 운영 모드 단계 전환** (App Check 의 monitoring → enforce) — rules deploy 사이클 없이 운영자가 트래픽 보면서 결정 가능. ADR 008 의 RSVP 정적 정책과 결의 다름

**ADR 007 update 확장의 의미** — delete 만 다루던 기존 결정을 update 까지 확장하면서 같은 트레이드오프 모델 일관 유지. 이건 ADR 의 "재방문" 패턴 — 새 ADR 를 쓰지 않고 기존을 갱신해 결정 history 를 한 곳에 모음.

## 3. 발견된 기술적 함정 7 종

이번 세션에 발화한 production-only 또는 React 19 / Next.js 16 / iOS Safari 함정을 명시 — 이후 같은 영역 작업 시 회피 자산.

### 3-1. `useSyncExternalStore` subscribe 안 동기 `callback()` 금지

**증상**: React 가 infinite loop 위험으로 bail-out → 컴포넌트 freeze (DDayBadge 카운트다운 화면에 안 나옴).

**원인**: subscribe 함수 안에서 callback 을 동기 호출하면 React 가 무한 갱신으로 인식.

**해결**: callback 동기 호출 제거. setInterval tick 으로만 callback 발화.

**더 깊은 결**: production 빌드에서 useSyncExternalStore 의 hydration transition 자체가 일관되지 않은 사례 발견 → **useEffect + useState + setInterval 직접 패턴으로 전환** (commit `88eb923`). React 19 + Next.js 16 turbopack production 의 hydration phase 에서 client snapshot 반영이 안정적이지 않음. dev 단계 검증으로 못 잡음.

**자산**: clock-like 상태는 useSyncExternalStore 보다 useEffect + useState + setInterval 이 production 안전.

### 3-2. `next/script` `onReady` 가 production build 에서 일관되게 fire 안 됨

**증상**: Kakao Maps 가 production 에서 안 보임 (script preloaded but not executed).

**원인**: `next/script` 의 `onReady` callback 이 production 빌드에서 일관되지 않게 발화. dev 에선 정상.

**해결**: `next/script` 폐기, `useEffect` 안에서 직접 `document.createElement('script')` + `script.onload` 패턴.

**자산**: 외부 SDK 통합 시 `next/script` 보다 직접 script 태그가 production 안정적.

### 3-3. Protocol-relative URL `//` 의 dev/HTTP 환경 한계

**증상**: dev 서버 (HTTP) 에서 Kakao Maps SDK script 가 `http://dapi.kakao.com/...` 로 떨어져 mixed content / preload 미실행 경고.

**원인**: `//dapi.kakao.com/...` 가 페이지 protocol 따라감 — production HTTPS 에선 `https://`, dev HTTP 에선 `http://`.

**해결**: 명시적 `https://` 프로토콜 사용.

**자산**: protocol-relative URL 은 dev/production 변수 줄이려면 명시 권장.

### 3-4. Kakao Maps SDK 의 별도 도메인 검증 (Share SDK 와 다른 경로)

**증상**: production 도메인 에서 Maps SDK 가 401 AccessDeniedError 반환 (Share SDK 는 정상 동작).

**원인**: Share SDK 는 `t1.kakaocdn.net` 정적 CDN 파일 (origin 체크 없음) + `Kakao.init()` 의 origin 검증으로 동작. Maps SDK 는 `dapi.kakao.com` 의 dynamic API 가 매 요청 origin 검증.

**해결**: Kakao Developer Console > 플랫폼 > Web 에 production 도메인 정확히 등록 (HTTPS 프로토콜, 슬래시·포트 없이).

**자산**: 같은 JavaScript 키여도 두 SDK 가 검증 경로 다름. `.claude/rules/kakao-sdk.md` 에 명시.

### 3-5. iOS Safari `audio.volume` read-only

**증상**: BGM fade-in/out 이 iOS 에서 동작 안 함.

**원인**: Apple 정책 — JavaScript 가 audio volume 변경 못 함 (사용자 하드웨어 버튼만).

**해결**: 우회 불가. 데스크톱·Android 만 fade 효과 + iOS 는 즉시 100% 볼륨으로 시작.

**자산**: iOS audio API 는 read-only 영역 다수. fade 같은 기능은 best-effort.

### 3-6. iOS Safari `preload="auto"` 무시

**증상**: BGM 재생 버튼 누르면 readyState 1 (HAVE_METADATA) 에서 정체. play() Promise 가 무한 pending.

**원인**: iOS 가 모바일 데이터 절약 차원에서 audio data 를 user gesture 까지 fetch 안 함.

**해결**: click handler 안에서 `new Audio(src)` 동적 생성 + `audio.play()` 호출. user gesture 안에서 생성된 Audio 는 iOS 가 "사용자 트리거 미디어" 로 인식해 fetch + play 한 번에 트리거.

**자산**: iOS audio 는 JSX `<audio>` 태그보다 click handler 안에서 동적 `new Audio()` 생성이 호환성 높음.

### 3-7. Vercel preview vs production 차이 + dev 환경 변수

**증상**: dev (HTTP + LAN IP `192.168.x.x`) 에서 BGM·카카오 SDK 다수 동작 안 함. Production 에선 정상.

**원인**: dev 환경의 변수 (HTTP / LAN IP / Next.js dev HMR / iPhone Safari 캐시) 종합. production HTTPS 에선 다 해결.

**해결**: dev 환경 디버깅 ROI 음수 수용 → **production Vercel preview 직행**. 진짜 target environment 검수.

**자산**: 모바일 Safari 검증은 production HTTPS 에서만. dev 환경 BGM/Kakao SDK 검증은 시간 낭비.

**채택 근거**: 비개발자 진입 마찰 90% 해소 (GitHub·Vercel 계정만) + 우리 인프라 0 (OAuth + API 통합) + OSS 정체성 (사용자 본인 GitHub 에 코드 생성, 우리 데이터 보관 안 함).
