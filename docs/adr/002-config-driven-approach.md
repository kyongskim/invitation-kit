# ADR 002 · Config-driven 아키텍처

- 상태: Accepted
- 날짜: 2026-04-23
- 관련: `invitation.config.ts`, `CLAUDE.md` (핵심 원칙 1)

---

## 맥락 (Context)

경쟁 레포를 분석한 결과 두 갈래로 나뉘었다.

- **국내 OSS 청첩장**: 컴포넌트 JSX 안에 신랑/신부 이름, 예식일, 계좌번호가 **하드코딩**되어 있다. Fork한 사람은 여러 파일을 뒤져가며 수정해야 하고, 메인 코드와 사용자 데이터가 섞여 업스트림 동기화가 어렵다.
- **해외 OSS**(wzulfikar 등): `config.js` 같은 단일 진입점이 있지만 한국 결혼식 필수 기능(카카오 공유, 한국 계좌, 지도 딥링크)이 없다.

Primary 타깃은 "주니어~시니어 프론트엔드 개발자"이고, Stretch 타깃은 "Fork만 해서 쓰고 싶은 비전공 개발자"다. 후자를 배제하지 않으려면 **수정 지점이 한 곳**이어야 한다.

또한 향후 v1.1+에서 구상 중인 **웹 기반 config 편집기**가 성립하려면, 사용자 데이터가 정형화된 하나의 스키마로 떨어져 있어야 한다.

## 결정 (Decision)

사용자가 직접 편집하는 값은 **모두 `invitation.config.ts` 한 파일**에 모은다.

- 단일 진입점: 프로젝트 루트의 `invitation.config.ts`
- 타입 안전성: `as const` + TypeScript 스키마로 자동완성과 컴파일 타임 검증 제공
- 컴포넌트 규약:
  - 섹션 컴포넌트는 config 값을 **props** 또는 최상단 **import**로만 받는다.
  - JSX 안에 개인정보(이름, 날짜, 계좌번호, 전화번호, 주소 등)를 **직접 쓰지 않는다**.
- 신규 기능은 먼저 config 스키마에 필드를 정의한 뒤 컴포넌트에서 소비한다.
- 스키마 변경(필드 제거, 이름 변경, 타입 변경)은 breaking change이므로 ADR로 기록하고 `CHANGELOG.md`에 남긴다.

## 결과 (Consequences)

**긍정:**

- Fork한 사용자는 단 한 파일만 수정해도 자신의 청첩장이 완성된다 → "5분 배포" 목표와 직접 정렬.
- 업스트림 업데이트 시 merge conflict가 `invitation.config.ts` 한 파일 기준으로만 발생해 관리가 쉽다.
- TypeScript 스키마가 곧 문서 역할을 한다. 사용자는 IDE 자동완성만으로도 어떤 필드가 있는지 파악 가능.
- 향후 웹 에디터는 이 스키마를 폼으로 렌더링하기만 하면 된다.

**부정 / 주의:**

- 사용자가 **최소한의 TypeScript 문법**(객체, 배열, 문자열)은 이해해야 한다. 비개발자 타깃은 v1.1+ 웹 에디터 이전에는 장벽이 있다.
- 스키마가 커질수록 **파일이 비대**해진다. 섹션별 분리(`config/gallery.ts` 등)는 지금 도입하지 않는다 — "한 파일이 곧 진실"이라는 단순함이 먼저. 임계점(500줄 이상)에 도달하면 별도 ADR로 재검토.
- 테마/인터랙션 같은 **동적 behavior**는 config로 표현하기 어렵다. config는 **데이터**만 담고, behavior는 theme 파일과 컴포넌트에 둔다.
- 공개 레포 전제에서 사용자가 실수로 `invitation.config.ts`를 커밋할 수 있다 → 샘플은 `invitation.config.example.ts`로 제공하고 실제 파일은 `.gitignore`로 보호하는 패턴은 **도입하지 않는다**. 사용자의 Fork는 본인 데이터가 들어간 사본이고 이게 정상 사용 방식이기 때문. 대신 README에 "개인정보 노출 주의" 경고를 명시한다.

## 후속 작업

- `invitation.config.ts`의 필드별 주석에 "필수/선택", "예시", "주의사항" 명시
- 섹션 컴포넌트 작성 시 config import 규약을 `.claude/rules/section-component.md`에 정리
- v0.1.0 릴리스 전에 스키마 동결. 이후 변경은 semver major 규칙 적용
